## 一、整体架构

```
┌────────────┐  HTTPS  ┌──────────────────────────┐
│ 6 台员工电脑 │ ──────► │  阿里云香港轻量服务器        │
│ Tauri 客户端 │         │  Caddy(443) → :8787       │
└──────┬──────┘         │  backend-gateway           │
       │                └────────────┬───────────────┘
       │ 直连                        │
       ▼                             ├──► 云雾/Pockgo/APIMart 生图 API
  ┌──────────┐                       ├──► Supabase（service_role 管理操作）
  │ Supabase │                       └──► 阿里云 OSS（图片归档）
  │ Auth/DB  │
```

**关键划分：**
- Supabase Auth/DB → 客户端**直连**（anon key，受 RLS 保护）
- 生图 API、OSS 上传、admin 创建账号 → **必须经过云端网关**（敏感密钥只在服务器）

## 二、云服务器选型说明

### 2.1 为什么选阿里云轻量应用服务器香港地域

| 对比项 | 阿里云轻量香港 | 阿里云轻量大陆 | 腾讯云轻量香港 |
|--------|--------------|--------------|--------------|
| 备案要求 | **无需备案** | 需要 ICP 备案 | 无需备案 |
| 访问 Supabase 延迟 | ~50ms（AWS 同区） | ~150ms | ~60ms |
| 访问生图 API 延迟 | 低（国际出口好） | 受限 | 低 |
| 月费（2C2G） | **¥56** | ¥45 | ¥60 |
| 流量费 | **无固定流量，不收流量费** | 含流量包 | 含流量包 |

**结论：** 香港地域无需备案、延迟低、流量免费，是本系统最优选择。

### 2.2 配置选型：为什么选 2C2G

本系统的网关是**纯 IO 密集型**（转发 HTTP 请求、上传 OSS），不做本地计算：

| 资源 | 实际消耗 | 2C2G 是否够用 |
|------|---------|--------------|
| CPU | 转发请求，峰值 < 20% | 够用 |
| 内存 | 网关进程约 50-100MB，加 OS 约 500MB | 够用（配置了 2GB swap 兜底） |
| 磁盘 | 代码 + 编译缓存约 3GB，日志约 100MB/月 | 够用（40GB） |
| 带宽 | 6 人并发，每张图约 500KB，峰值 < 5Mbps | 够用（200Mbps 峰值） |

**日生 1000 张图的估算：**
- 每张图上传 OSS 约 500KB → 每日出流量约 500MB
- 200Mbps 带宽，500MB 流量约 20 秒传完，完全不是瓶颈
- 内存峰值（6 人同时生图）约 300MB，远低于 2GB

> 如果未来扩展到 20+ 人并发或日生 5000 张，建议升级到 2C4G（约 ¥100/月）。

### 2.3 操作系统选择

选择 **Alibaba Cloud Linux 3**（基于 RHEL 9 / OpenAnolis）：
- 阿里云官方维护，内核优化好
- 使用 `dnf` 包管理器
- 长期支持，安全更新及时

## 三、第三方服务资源消耗分析

### 3.1 Supabase Free Tier 够用吗

Supabase 免费套餐限制与本系统实际用量对比：

| 资源 | Free Tier 限制 | 本系统实际用量 | 是否够用 |
|------|--------------|--------------|---------|
| 数据库存储 | 500MB | 用户表 + 生图日志，6 人 × 1000 条/天 × 3 天保留 ≈ **2MB** | **远够** |
| 数据库行数 | 无限制 | — | — |
| Auth 用户数 | 50,000 | 6 人 | **远够** |
| 每月 API 请求 | 500 万次 | 6 人 × 1000 次/天 × 30 天 = **18 万次** | **远够（3.6%）** |
| 出口流量（Egress） | 5GB/月 | 登录验证 + 数据库查询，约 **200MB/月** | **远够（4%）** |
| 并发连接 | 60 | 6 人，峰值 < 10 | **远够** |
| 项目数 | 2 个 | 1 个 | 够用 |
| 项目暂停 | 7 天不活跃自动暂停 | 每天都有使用，不会暂停 | 无影响 |

**结论：Supabase Free Tier 对本系统完全够用，无需付费升级。**

> 注意：generation_logs 表保留期设为 3 天（已配置定时清理），避免日志无限增长。

### 3.2 阿里云 OSS 消耗估算

OSS 费用由三部分组成：存储费 + 请求费 + 流量费。

**存储费：**
- 每张图压缩后约 200-400KB，取均值 300KB
- 日生 1000 张 × 300KB = 300MB/天
- 月存储增量：300MB × 30 = **9GB/月**
- 存储单价：约 ¥0.12/GB/月
- 月存储费：9GB × ¥0.12 ≈ **¥1.1/月**（累计存储，半年后约 ¥6/月）

**请求费：**
- PutObject 请求：1000 次/天 × 30 天 = 3 万次/月
- 请求单价：¥0.01/万次
- 月请求费：**< ¥0.1/月**

**流量费（内网访问）：**
- 服务器在阿里云香港，OSS 也在阿里云 → 走**内网传输，免费**
- 员工下载图片走 OSS 外网 URL → 约 9GB/月 × ¥0.5/GB ≈ **¥4.5/月**

**OSS 月费合计：约 ¥6-15/月**（随存储累积增长）

> 建议：设置 OSS 生命周期规则，超过 90 天的图片自动转为低频存储（费用降低 60%）。

### 3.3 整体月度成本汇总


## 十一、故障排查

| 现象 | 排查命令 | 常见原因 |
|------|---------|---------|
| 健康检查 502 | `systemctl status csgh-backend-gateway` | gateway.env 缺密钥 / 端口被占 |
| HTTPS 证书失败 | `journalctl -u caddy -e` | DNS 未生效 / 80 端口被挡 |
| 客户端登录 401 | 检查两端 VITE_SUPABASE_URL 是否一致 | 指向不同 Supabase 项目 |
| 生图 502/超时 | `journalctl -u csgh-backend-gateway --since "10 min ago"` | 上游 API Key 失效 |
| OOM 重启 | `dmesg -T \| grep -i killed` | 内存不足，检查 swap |
| 客户端启动闪退 | Windows 事件查看器 → 应用程序日志 | 构建时未设置环境变量 |
