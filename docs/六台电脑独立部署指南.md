# 六台电脑独立部署指南（无网关 · 本地直连模式）

> 适用场景：6 台 Windows 电脑各自独立安装并使用，**不依赖任何中心服务器 / 网关电脑**。每台电脑直接通过本机 Rust 进程调用云雾生图、阿里云 OSS、Supabase。
>
> 数据共享：所有电脑登录同一个 Supabase 项目，账号体系、历史记录、管理后台数据自动互通。

---

## 0. 部署架构总览

```
                        ┌──────────────────┐
                        │  云端共享层       │
                        │  · Supabase      │ ← 账号 / 历史 / 管理后台
                        │  · 阿里云 OSS    │ ← 生成图片永久存储
                        │  · 云雾 / Pockgo │ ← 5 条生图线路
                        │    / APIMart 等  │
                        └────────▲─────────┘
                                 │ 各台电脑独立 HTTPS 直连
       ┌───────┬───────┬─────────┼─────────┬───────┬───────┐
    [电脑1]  [电脑2]  [电脑3]  [电脑4]  [电脑5]  [电脑6]
    桌面应用 桌面应用 桌面应用 桌面应用 桌面应用 桌面应用
    +.env.local +.env.local +.env.local +.env.local +.env.local +.env.local
```

每台电脑 = 一份桌面安装包 + 一份 `.env.local`，独立、对等、互不依赖。

## 1. 前置条件

### 1.1 各台电脑硬件 / 系统要求
- Windows 10 1809+ 或 Windows 11
- 内存 ≥ 8 GB（生图过程会临时占用 1–2 GB）
- 磁盘 ≥ 10 GB 可用（应用 + 缓存图片）
- 必须能稳定访问外网（Supabase / OSS / 生图接口）

### 1.2 你需要先准备好的云端账号
| 项 | 用途 | 准备方式 |
|---|---|---|
| Supabase 项目 | 账号 / 历史 / 管理后台 | 在 https://supabase.com 创建，按本仓库 `supabase/README.md` 完成初始化 |
| 阿里云 OSS Bucket | 生成图片存储 | 阿里云控制台创建 Bucket，开通 AccessKey |
| 5 条生图线路的 API Key | 实际生图 | 云雾 / VectorEngine / Pockgo / APIMart 各家平台开通（不必全部，至少要 1 条线路 Key） |

### 1.3 你需要做出的密钥分发选择

| 方案 | 描述 | 推荐度 |
|---|---|---|
| **A：6 台同配** | 6 台电脑都放完整 `.env.local`，含 `SUPABASE_SERVICE_ROLE_KEY` | ⚠️ 简单但风险大 |
| **B：1 管理 + 5 员工** | 管理员 1 台放完整 `.env.local`；员工 5 台不放 `SUPABASE_SERVICE_ROLE_KEY`，建账号在管理员机或 Supabase 后台完成 | ✅ 推荐 |

> **service_role key 一旦下发到员工机，等同于放弃 Supabase 的全部行级权限保护**。后续遇到员工离职，必须立即在 Supabase Dashboard 轮换该 Key 并重新分发到留下的电脑。

## 2. 一次性云端初始化（只做一次）

由系统管理员在自己电脑完成。已做过可跳过。

### 2.1 Supabase 项目初始化
1. 登录 https://supabase.com → **New project**，名字任填
   - Database Password 自己设置并妥善保存
   - Region 选 `Northeast Asia (Tokyo)` 或 `East Asia (Hong Kong)`
2. 进入项目 → 左侧 **SQL Editor** → New query
3. 粘贴本仓库 `supabase/schema.sql` 全文并 Run
4. 依次粘贴并 Run `supabase/migrations/` 目录下**全部** `.sql` 文件（按文件名时间顺序）
5. 创建第一个管理员账号：
   - **Authentication → Users → Add user → Create new user**
   - 勾选 *Auto Confirm User*，记下邮箱与密码
   - 回到 SQL Editor 运行：
     ```sql
     update public.profiles
        set role = 'admin'
      where id = (select id from auth.users where email = '管理员邮箱');
     ```
6. **Project Settings → API** 复制以下三个值，下一步要用：
   - `Project URL` → 后面写入 `VITE_SUPABASE_URL`
   - `anon public key` → 后面写入 `VITE_SUPABASE_ANON_KEY`
   - `service_role key`（只复制一次，妥善保管）→ 后面写入 `SUPABASE_SERVICE_ROLE_KEY`

### 2.2 阿里云 OSS 准备
1. 控制台 → 对象存储 OSS → 创建 Bucket（建议 `oss-cn-hangzhou`，私有读写）
2. 控制台 → 访问控制 RAM → 创建子用户，授权 `AliyunOSSFullAccess`（或更精细的 Bucket 级策略），生成 AccessKey ID + Secret
3. 记录：`ALI_OSS_REGION` / `ALI_OSS_BUCKET` / `ALI_OSS_ACCESS_KEY_ID` / `ALI_OSS_ACCESS_KEY_SECRET`

### 2.3 生图线路 API Key
按你实际使用的线路准备（至少需要 1 条）：

| 环境变量 | 对应线路 | 是否必需 |
|---|---|---|
| `IMAGE_2_API_KEY` | 线路 1（云雾 image-2） | 推荐都配 |
| `IMAGE_2_LINE2_API_KEY` | 线路 2（云雾编辑） | 用线路 2 才需要 |
| `VECTORENGINE_IMAGE_2_API_KEY` | 线路 3（VectorEngine） | 用线路 3 才需要 |
| `POCKGO_IMAGE_2_API_KEY` | 线路 4（Pockgo） | 用线路 4 才需要 |
| `APIMART_IMAGE_2_API_KEY` | 线路 5（APIMart） | 用线路 5 才需要 |

## 3. 准备一份"安装母盘"（在管理员电脑做一次，6 台共用）

这一步做完一次后，你会得到：
- 一个 Windows 安装包：`csgh-image-studio_2.0.0_x64-setup.exe`（或同目录的 `.msi`）
- 一份 `.env.local` 模板（不放进安装包，单独通过 U 盘 / 加密通道分发）

### 3.1 在管理员电脑装好开发环境
仅母盘机器需要，员工机不需要。
- Node.js ≥ 18（建议 20 LTS）：https://nodejs.org
- Rust stable + Cargo：https://rustup.rs
- Microsoft Visual Studio 2022 Build Tools，勾选 *使用 C++ 的桌面开发*
- WebView2 Runtime：https://developer.microsoft.com/microsoft-edge/webview2/

### 3.2 拉项目并安装依赖
```powershell
cd F:\christescuobermeyer-max\image-2生图系统
npm install
```

### 3.3 准备临时 `.env.local`（仅供构建用）
构建时**不要**设置 `VITE_BACKEND_GATEWAY_URL`，让应用走本地直连模式。

最小可运行的 `.env.local`（构建阶段只是为了让 Vite 注入 Supabase URL / anon key 这两个 `VITE_` 前缀变量；其余 Key 是运行时由 Rust 读取，可以暂时留空）：

```env
VITE_SUPABASE_URL=https://你的项目.supabase.co
VITE_SUPABASE_ANON_KEY=eyJhbGciOi...你的_anon_key
# 注意：不要写 VITE_BACKEND_GATEWAY_URL，让应用走本地直连
```

### 3.4 检查 Tauri 环境（自动选择 VS C++ 工具链）
```powershell
npm run tauri:doctor
```

### 3.5 打包安装程序
```powershell
npm run tauri:build
```

构建产物在：
- `src-tauri\target\release\bundle\nsis\csgh-image-studio_2.0.0_x64-setup.exe`
- `src-tauri\target\release\bundle\msi\csgh-image-studio_2.0.0_x64_zh-CN.msi`

任选其一作为分发安装包。

## 4. 在每台员工电脑安装（重复 6 次，每台 ≈ 5–10 分钟）

### 4.1 安装 WebView2 Runtime
Windows 11 已内置；Windows 10 / Server 需要先装：
https://developer.microsoft.com/microsoft-edge/webview2/

### 4.2 双击安装包
运行 `csgh-image-studio_2.0.0_x64-setup.exe`，按向导安装即可。默认安装路径：
```
C:\Users\<用户名>\AppData\Local\呈尚策划美工生图系统PRO\
```

### 4.3 写入该机的 `.env.local`
**关键步骤**。Rust 进程启动时会从安装目录或当前工作目录的 `.env.local` 读取所有运行时密钥。

#### 安装目录定位
应用 `.env.local` 必须放在 **应用可执行文件所在目录**，即上一步默认安装路径下的根目录（与 `呈尚策划美工生图系统PRO.exe` 同级）。

> 实际加载顺序由 [src-tauri/src/env_config.rs](../src-tauri/src/env_config.rs) 控制：dotenvy 会从当前工作目录向上查找；为最稳妥，**直接把 `.env.local` 放到与 .exe 同级的安装目录**。

#### 方案 A：6 台同配（最简单）

每台电脑 `.env.local` 内容完全相同：

```env
# Supabase 共享层
VITE_SUPABASE_URL=https://你的项目.supabase.co
VITE_SUPABASE_ANON_KEY=eyJhbGciOi...你的_anon_key
SUPABASE_SERVICE_ROLE_KEY=eyJhbGciOi...你的_service_role_key

# 生图线路（按需填，至少 1 条）
IMAGE_2_API_KEY=sk-...线路1key
IMAGE_2_LINE2_API_KEY=sk-...线路2key
VECTORENGINE_IMAGE_2_API_KEY=sk-...线路3key
POCKGO_IMAGE_2_API_KEY=sk-...线路4key
APIMART_IMAGE_2_API_KEY=sk-...线路5key

# 阿里云 OSS
ALI_OSS_REGION=oss-cn-hangzhou
ALI_OSS_BUCKET=你的bucket
ALI_OSS_ACCESS_KEY_ID=LTAI...
ALI_OSS_ACCESS_KEY_SECRET=...secret
```

#### 方案 B（推荐）：1 管理员 + 5 员工

**管理员电脑 `.env.local`**：与方案 A 完全一致（含 `SUPABASE_SERVICE_ROLE_KEY`）。可在应用内"后台管理 → 新增账号"创建员工账号。

**员工 5 台电脑 `.env.local`**：删掉 `SUPABASE_SERVICE_ROLE_KEY` 那一行，其余相同。员工机内置的"新增账号"会失败，但生图、OSS、历史、登录全部可用，符合预期最小权限。

```env
# 与上面一致，但仅删除以下一行：
# SUPABASE_SERVICE_ROLE_KEY=...   ← 员工机不放此行
```

### 4.4 写入文件后做一次健康检查
1. 双击启动应用
2. 用第 2.1 步创建的管理员账号或员工账号登录
3. 进入"店招/头像"页，上传一张 PNG，选线路 1，点"开始生成"
4. 等到出图（约 1–6 分钟），右下角"下载"测试本地保存
5. 进入"历史"页，确认刚才的记录已写入云端
6. 管理员机：进入"后台管理"，确认能看到当前用户

如有报错：
- "登录态已失效" → `VITE_SUPABASE_URL` 或 anon key 写错
- "未找到 IMAGE_2_API_KEY" → 该线路 Key 没填或拼错
- OSS 上传失败 → AccessKey 权限或 Bucket region 不对

### 4.5 桌面快捷方式
安装包默认在桌面和开始菜单创建快捷方式，无需额外操作。

## 5. 关键运维流程

### 5.1 新增 / 升级生图线路 Key
某条线路换密钥时，需要把新 Key 同步到所有电脑：

```
推荐做法：
1. 把新 .env.local 模板放到一个加密 U 盘 / 内部加密邮箱
2. 由管理员到每台电脑替换文件
3. 替换后无需重装，重新启动应用即可生效（应用启动时一次性读 .env.local）
```

### 5.2 升级应用版本
```
1. 管理员电脑拉取最新代码 → npm install → npm run tauri:build
2. 把新生成的 .exe / .msi 通过内网共享 / U 盘分发到 6 台电脑
3. 每台电脑双击安装包覆盖安装（不影响 .env.local）
4. 启动应用确认版本号
```

### 5.3 创建新员工账号
- 方案 A：任意一台都可用应用内"后台管理 → 新增账号"
- 方案 B（推荐）：
  - 管理员机用应用内"新增账号"功能
  - 或在 Supabase Dashboard → **Authentication → Users → Add user**（勾选 *Auto Confirm User*），再到 SQL Editor 把 role 留空（默认 `user`）
  - 把账号密码发给员工，员工到任意电脑登录

### 5.4 注销 / 离职员工
1. Supabase Dashboard → **Authentication → Users** → 找到该员工 → 删除或禁用
2. 如果该员工接触过任何一台员工机的 `.env.local`，**强烈建议轮换全部密钥**：
   - Supabase service_role key（Dashboard → API → Reset）
   - 阿里云 OSS AccessKey（RAM 控制台禁用旧的、生成新的）
   - 各生图线路 Key（联系对应平台轮换）
   - 同步新 Key 到剩余电脑的 `.env.local`

### 5.5 历史数据治理
- 应用每天会调用 `cleanup_expired_generation_logs()`，自动删除 7 天前的生图记录
- OSS 端通过签名 URL 7 天过期，超期图片自动失效
- 如需清空所有历史，在 Supabase SQL Editor 执行：
  ```sql
  truncate table public.generation_logs;
  truncate table public.login_logs;
  ```

## 6. 安全加固清单（强烈建议在量产分发前完成）

> 这些项与 6 台独立部署模式直接相关，源自前一次代码审查。

| 编号 | 项 | 状态 | 说明 |
|---|---|---|---|
| S1 | `.env.local` 文件权限设为仅当前用户可读 | 必做 | 右键 → 属性 → 安全 → 仅保留当前用户读权限 |
| S2 | 不要把 `.env.local` 通过明文邮件 / IM 发送 | 必做 | 推荐加密 U 盘或受控网盘加密压缩包 |
| S3 | 关闭安装目录在网络共享中的可见性 | 建议 | 防止局域网内其他设备直接拉取 .env.local |
| S4 | 员工机不要放 `SUPABASE_SERVICE_ROLE_KEY` | 强烈建议 | 即方案 B |
| S5 | 阿里云 OSS RAM 子账号收紧到该 Bucket 级权限 | 建议 | 不要给 `AliyunOSSFullAccess`，改为 Bucket 级策略 |
| S6 | 阿里云 OSS AccessKey 设置 IP 白名单 | 可选 | 如 6 台电脑公网 IP 固定 |
| S7 | 给所有员工 Supabase 账号开启 MFA | 建议 | Supabase Dashboard → Authentication → Policies |
| S8 | 6 台电脑都开启磁盘加密（BitLocker） | 强烈建议 | 防止设备遗失 / 被盗导致 .env.local 泄漏 |
| S9 | 定期（如每季度）轮换全部 API Key 一次 | 建议 | 即便没有员工变动也要做 |

## 7. 故障速查

| 现象 | 可能原因 | 处理 |
|---|---|---|
| 启动后停在"正在恢复登录状态…" | Supabase URL 不通 / 网络问题 | 检查网络；检查 `VITE_SUPABASE_URL` |
| 登录报"邮箱或密码错误" | 实际错误 / 用户未 Auto Confirm | Supabase Dashboard 检查用户状态 |
| 生图返回"未找到 IMAGE_2_API_KEY" | `.env.local` 路径不对 / 漏字段 | 确认 `.env.local` 与 .exe 同级、字段名完全一致 |
| 生图卡在"生成中…"5 分钟以上 | 上游线路本身慢 / 网络抖动 | 应用已设 300s 超时；可手动取消重试，或换一条线路 |
| OSS 上传失败 | AccessKey 权限 / region 错 / Bucket 不存在 | 检查 RAM 权限、`ALI_OSS_REGION` 是否与 Bucket 一致 |
| 后台管理看不到用户 | 当前账号 role 不是 admin | 在 SQL Editor 中 `update public.profiles set role='admin' where ...` |
| 应用内"新增账号"报权限错 | 该机 `.env.local` 没有 `SUPABASE_SERVICE_ROLE_KEY` | 这是方案 B 的预期；改用 Supabase Dashboard 创建 |
| 历史不显示某张图 | 已超过 7 天 OSS 链接过期 | 预期行为，无法恢复 |

## 9. 与"网关模式"的对比（如果未来想切换）

如果将来你改主意，想从"6 台独立"切回"1 台网关 + 6 台轻客户端"模式：

| 维度 | 当前模式（6 台独立） | 网关模式 |
|---|---|---|
| 密钥分布 | 6 份 | 1 份 |
| 离职 / 设备丢失影响 | 必须轮换 6 台 Key | 只需把员工机重新指向网关即可 |
| 员工机性能要求 | 需要 Rust 进程跑生图 / OSS | 仅 UI，资源占用低 |
| 中心机宕机 | 不影响 | 6 台全停 |
| 配置复杂度 | 中 | 低（员工机只填一个网关 URL） |

切换方式：每台员工机的 `.env.local` 添加：
```env
VITE_BACKEND_GATEWAY_URL=http://中心机内网IP:8787
```
重新构建并安装即可，参见仓库 `docs/local-backend-gateway-windows.md`。

## 10. 一页速查（贴墙版）

```
6 台电脑独立部署 = 桌面安装包 + .env.local（每台 1 份）+ Supabase/OSS/生图 Key（云端共享）

部署顺序：
1) 管理员一次性建好 Supabase + OSS + 申请生图 Key
2) 管理员一台机 npm run tauri:build → 得到 .exe / .msi 安装包
3) 把安装包 + .env.local 模板拷到每台电脑
4) 每台电脑：装 WebView2 → 装应用 → 把 .env.local 放到与 .exe 同级目录
5) 启动 → 登录 → 生图测试 → 完成

最大风险：API Key 散落 6 台电脑
最小化做法（方案 B）：5 台员工机不放 service_role key + 全部启用 BitLocker
```
